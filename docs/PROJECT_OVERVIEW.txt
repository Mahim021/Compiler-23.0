================================================================================
                            1. PROJECT DESCRIPTION
================================================================================

This project implements a LEXICAL ANALYZER (also called a tokenizer or scanner)
using Flex - a tool for generating lexical analyzers. The lexer reads source
code and breaks it down into meaningful units called "tokens".

PURPOSE:
- Learn compiler design fundamentals
- Understand tokenization (first phase of compilation)
- Practice using Flex for pattern matching
- Prepare foundation for parser integration (Bison)

                    15. IMPLEMENTATION CHALLENGES SOLVED
================================================================================

CHALLENGE 1: Keyword vs Identifier Conflict
  Problem: "if" being recognized as identifier instead of keyword
  Solution: Placed keyword rules BEFORE identifier rule in flex.l
  Lesson: Rule order matters in Flex - most specific first!

CHALLENGE 2: Multi-character Operator Parsing
  Problem: ">=" being tokenized as ">" and "=" separately
  Solution: Listed >=, <=, ==, != BEFORE single-character operators
  Lesson: Flex uses longest match, but needs to see longer patterns first

CHALLENGE 3: Multi-line Comment Handling
  Problem: Comments spanning multiple lines not properly handled
  Solution: Implemented state machine with %x COMMENT
  Code:
    "/*"              { BEGIN(COMMENT); }
    <COMMENT>"*/"     { BEGIN(INITIAL); }
    <COMMENT>.        { /* ignore */ }
    <COMMENT>\n       { /* yylineno auto-increments */ }

CHALLENGE 4: String Literals with Spaces
  Problem: Strings like "Hello, World" treated as separate tokens
  Solution: Pattern \"[^\"]*\" matches everything between quotes
  Lesson: Character class [^\"] means "anything except quote"

CHALLENGE 5: Line Number Tracking
  Problem: Manual tracking (line_number++) error-prone and tedious
  Solution: Used %option yylineno - automatic and reliable
  Benefit: Cleaner code, accurate counting even with comments

CHALLENGE 6: Comment Line Counting
  Problem: Line numbers wrong after multi-line comments
  Solution: yylineno automatically increments on \n, even in <COMMENT> state
  Result: Accurate line numbers throughout file

CHALLENGE 7: File I/O Error Handling
  Problem: Program crashes if input.txt doesn't exist
  Solution: Added proper error checking:
    if (!input_file) {
        printf("ERROR: Cannot open input.txt\n");
        return 1;
    }

CHALLENGE 8: Distinguishing Different Delimiters
  Problem: Three bracket types - when to use which?
  Solution: Designed semantic distinction:
    () - conditions and parameters
    [] - function bodies 
    {} - control flow blocks
  Documented in comments for consistency

CHALLENGE 9: Testing Complex Nested Structures

DESIGN PRINCIPLES YOU FOLLOWED:

1. READABILITY FIRST
   - "loop" instead of cryptic "for(;;)" syntax
   - "to" keyword makes range clear: "0 to n"
   - "elif" more concise than "else if"

2. VISUAL DISTINCTION
   - Three delimiter types with semantic meaning:
     () = conditions/parameters (input/grouping)
     [] = function definitions (permanent structure)
     {} = control flow blocks (temporary scope)

3. FLEXIBILITY
   - Optional semicolons (Python-like freedom)
   - Dynamic typing (no int, string, etc. needed)
   - String concatenation with familiar + operator

4. SIMPLICITY
   - No pointer syntax (*ptr, &ref)
   - No complex type system (int*, char[], etc.)
   - Built-in I/O (read, write) - no printf complexity

5. MODERN FEATURES
   - Clean loop syntax with explicit step: loop(i: +1: 0 to n)
   - elif construct (like Python, Ruby)
   - Multi-line comments for documentation

6. CONSISTENCY
   - All keywords lowercase
   - All built-in functions lowercase
   - Uniform delimiter usage

YOUR LANGUAGE COMPARED TO STANDARD LANGUAGES:

Feature          | Your Language  | C/C++        | Python      | Java
-----------------|----------------|--------------|-------------|-------------
Function Body    | []             | {}           | :+indent    | {}
Elif             | elif           | else if      | elif        | else if
Loop             | loop(){}       | for(){}      | for:        | for(){}
Range            | 0 to n         | i=0;i<n;i++  | range(0,n)  | i<n
Type Declaration | none (dynamic) | int x;       | none        | int x;
Semicolons       | optional       | required     | no          | required
Comments         | // and /* */   | // and /* */ | # and """   | // and /* */

UNIQUE INNOVATIONS YOU INTRODUCED:

✓ Loop step parameter: loop(i: +1: 0 to n){...}
  - Explicit increment/decrement (+1, -1, +2, etc.)
  - More readable than C's i++, i--, i+=2

✓ Reverse loops made easy: loop(i: -1: n-1 to -1){...}
  - No need to remember complex for loop syntax
  - Direction is clear from the step parameter

✓ Square brackets for functions: functionName()[...]
  - Visual distinction from control flow blocks {}
  - Makes function definitions stand out

✓ Read with prompt: read("Enter value: ":variable)
  - Combined prompt and input in one call
  - More convenient than separate print + scanf

✓ Check keyword (to be defined in parser phase)
  - Appears to be for assertions or validations
  - Example: loop(i: 1){...}check(0 to n)
